# My OpenCode Configuration

为 AI 编码助手提供结构化、可持续优化的开发方法论。

## 核心原则

1. **完整性优先** — 完整实现用户目标、完整验证结果、完整清理本次改动造成的副作用；不要把完整性误解为添加未请求功能。
2. **简洁优先** — 最小代码解决当前问题。不要为单次使用创建抽象，不添加未请求的灵活性或未来扩展。
3. **手术式修改** — 每一行改动都必须能追溯到用户请求。不要顺手重构、格式化、改注释或清理无关死代码。
4. **先搜索再构建** — 写代码之前，先搜索是否有现成的工具、库或模式解决同样的问题。
5. **测试驱动** — 先写测试，看它失败，再写最小代码使其通过。80%+ 测试覆盖率是底线。
6. **安全第一** — 验证所有输入，不硬编码密钥，做参数化查询。
7. **用户说了算** — AI 推荐方案，用户做最终决定。当你有强烈的建议但用户反对时，解释理由然后尊重选择。

## 关键工作流

收到任何开发任务时，按照四阶段流程工作：

```
用户: "帮我实现用户认证"
    ↓
/brainstorm → 设计探索 → 澄清问题 → 方案对比 → 设计文档 → 用户批准
    ↓
/plan → 任务分解 → 计划文档 → 用户确认
    ↓
/execute 1 → 实现(含TDD) → spec-compliance-review → code-review → 完成
/execute 2 → ...
    ↓
/finish → 验证测试 → 提交 → 合并/PR/清理
```

### 各阶段说明

**阶段 1: /brainstorm (设计探索)**
- 探索项目上下文 → 逐一提问澄清需求 → 提出 2-3 个方案
- 分段呈现设计 → 用户批准 → 保存设计文档到 `Project/{项目名称}/docs/designs/`
- **HARD-GATE**: 批准前禁止写任何代码

**阶段 2: /plan (计划制定)**
- 读取设计文档 → 拆分为 2-5 分钟可完成的独立任务
- 每个任务: 文件路径 + 测试要求 + 验证步骤 + 依赖关系
- 保存计划文档到 `Project/{项目名称}/docs/plans/`

**阶段 3: /execute <编号> (执行任务)**
- 每个任务走 TDD 流程: 先写测试 → 实现 → 审查
- 两阶段审查: spec-compliance (对照设计) → code-quality (代码质量)
- 自动连续执行，不暂停

**阶段 4: /finish (分支收尾)**
- 最终验证 → 代码整理 → 提交 → 提供交付选项
- 创建 PR / 合并 / 保留 / 清理

### 强制规则

**代码编写前：**
- 必须完成 brainstorming 并获得用户批准（适用于所有项目，无论多简单）
- 必须完成 writing-plans 并获得用户确认
- 必须显式识别假设、歧义和成功标准；不确定时先问，不要静默选择

**代码编写中：**
- 必须先写失败测试，再写实现代码（TDD）
- 禁止修改 linter/formatter 配置文件来绕过代码质量检查
- 只修改完成当前目标所必需的代码；禁止 drive-by refactor

**代码编写后：**
- 必须运行验证命令确认修改正确
- 必须进行代码审查（spec + quality 两阶段）
- 禁止提交存在安全隐患的代码（硬编码密钥、未验证的用户输入等）

## 技能路由表 & 命令

### 主工作流命令（按流程顺序）

| 命令 | 对应技能 | 何时使用 |
|------|---------|---------|
| `/brainstorm` | brainstorming | 任何新功能/创意/修改行为 — 设计探索第一步 |

| `/execute <N>` | execute-plans | 按计划逐个执行任务 |

| 命令 | 对应技能 | 何时使用 |
|------|---------|---------|
| `/quality` | code-quality-audit | 评估代码质量 — 自动检测项目类型并使用对应评估框架 |
| `/review` | code-review | 代码修改完成后 — 审查质量 |
| `/debug` | systematic-debugging | 遇到 Bug/测试失败/异常行为 |

### 学习与优化闭环

| 命令 | 功能 | 何时使用 |
|------|------|---------|
| `/learn` | 会话内提取模式 → 置信度进化 | 每次完成重要任务后 |
| `/learn-status` | 查看项目的学习积累状态 | 开始新会话时 |
| `/learn-evolve` | 聚类相关学习 → 升级为 rules/skills | 积累 ≥ 5 confirm 时 |
| `/insights` | 30天回顾 → 使用分析 + 摩擦识别 | 每周一次 |
| `/optimize` | 将 insights/quality 报告转化为优化 spec | `/insights` 或 `/quality` 之后 |
| `/quality` | 代码质量评估 → 评分报告 | 按需 |

### Spec 驱动的优化方法论

```
/insights 或 /quality → 诊断报告 (问题清单)
         ↓
    /optimize → optimization-spec skill
         ├─ 读取报告 → 提取可优化项
         ├─ 优先级矩阵 (影响×3 / 成本×1)
         ├─ 为 Top 3 编写优化 spec (问题→方案→预期→验证)
         └─ 终端状态 → 建议 /plan
              ↓
         /plan → writing-plans (拆解为实施任务)
              ↓
         /execute 1..N → 逐个执行 (含 TDD + 两阶段 review)
              ↓
         /finish → 验证优化效果 (重新运行 /quality 对比)
```

**核心纪律**:
- 每个优化必须有 spec — 不凭直觉改配置
- 可量化对比 — before/after 指标
- 渐进式 — 一次一个维度，逐个验证
- 可回滚 — 每个优化独立，出错可单独回滚

**复利 + 洞察 + 优化 三循环:**
```
/learn:    微观 — 会话内提取 → 跨会话确认 → 升级为配置
/insights: 宏观 — 30天回顾 → 识别摩擦 → 发现优化机会
/optimize: 行动 — 报告 → spec → plan → execute → finish → 验证
```

### OpenWolf-lite 自动机制（warning-only）

- `pre-read`: 同会话重复读取同一文件时告警；若 anatomy-lite 命中，提示文件摘要与估算 token。
- `pre-write`: 命中 Do-Not-Repeat 规则时告警；仅提醒，不阻断写入。
- `post-write`: 自动更新 `.opencode/state/openwolf-lite/anatomy-lite.json`。
- `session.idle`: 自动汇总会话到 `.opencode/state/openwolf-lite/session-ledger.json`。

## 技能自动触发机制

你必须主动识别当前场景并 invoke 对应技能。以下规则不可协商：

- 哪怕是 1% 的可能性一个技能适用，你就必须 invoke 它
- 如果技能存在且适用于当前任务，你没有选择，必须使用
- 在 invoke 技能之前，不要写任何代码，不要探索代码库

## 危险的合理化借口

以下想法意味着你正在试图绕过流程。识别它们，拒绝它们：

| 你的想法 | 事实 |
|---------|------|
| "这只是一个简单问题" | 问题也是任务。检查是否有适用的技能。 |
| "我先看看代码库再说" | 技能告诉你如何探索。先 invoke 技能。 |
| "不需要走正式流程" | 如果技能存在，就用它。简单的项目最容易因未经审视的假设而浪费工作。 |
| "我记得这个技能的内容" | 技能会更新。读当前版本，不要凭记忆。 |
| "技能太繁重了" | 简单的事情会因为跳过流程而变得复杂。使用技能。 |
| "我先做这一件事再说" | 在做任何事情之前先检查技能。 |
| "这感觉很有生产力" | 无序的行动浪费了时间。技能防止这种情况。 |

## Agent 编排

以下场景你应该主动使用子 Agent，不需要等待用户指示：

| 场景 | 启动 Agent | 原因 |
|------|-----------|------|
| 复杂功能、重构规划 | planner | 计划需要专门的分析和设计 |
| 架构决策 | planner | 需要系统性思考 |
| 刚完成一段代码 | reviewer | 需要独立视角的质量检查 |
| 安全相关的代码 | security-reviewer | 需要专门的安全知识 |
| Bug 调查 | debugger | 需要系统化的调试方法 |

## 编码风格

- 不可变性优先：尽可能创建新对象，不要修改现有对象
- 函数保持小巧（< 50 行），文件保持专注（< 800 行）
- 嵌套不超过 4 层
- 正确进行错误处理，不使用空的 catch 块
- 合理命名标识符，避免无意义的变量名

## 安全指南

在任何提交之前确保：
- 没有硬编码的密钥（API key、密码、token）
- 所有用户输入经过验证
- SQL 使用参数化查询防止注入
- HTML 输出经过转义防止 XSS
- 认证和授权机制正确
- 错误消息不泄露敏感信息

如发现安全问题：STOP → 使用 security-reviewer agent → 修复高危问题 → 轮换泄露的密钥 → 审查整个代码库的同类问题。
